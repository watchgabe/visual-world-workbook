## Summary

Phase 05.1 is a targeted remediation pass — existing section components are in place and functional (auto-save, field registration, routing all working), but several complex interactive features were deferred during Phase 5. This research has verified the exact behavior of each missing feature from the old HTML source files.

The work falls into three categories: (1) simple augmentations to existing components (AI generate buttons, color picker, Set My Date button), (2) component rewrites requiring new UI systems (Creator Analysis with dynamic cards, Mood Board gallery with localStorage, Typography with font preview/autocomplete), and (3) new sections that need to be created and registered (Cinematic Content, Content Blueprint).

The old HTML modules at `old/modules/` have been verified as complete canonical references. All AI prompts are extractable verbatim from those files. The `/api/claude` route, `useAutoSave`, `WorkshopInput`/`WorkshopTextarea`/`OptionSelector` components, and the `SECTION_REGISTRY` pattern are all in place and reusable for new features.

**Primary recommendation:** Work module by module, smallest to largest. Launch (simple AI buttons + date math) → Brand Foundation (audit scoring + AI buttons) → Content (angle fields + two new sections) → Visual World (complex UI systems last, risk-isolated).

### Pattern: AI Generate Button
Extracted from `trust-and-money.tsx` (the only existing example with AI generation):

```typescript
const [isGenerating, setIsGenerating] = useState<string | null>(null)

async function handleGenerate(fieldKey: string, contextFields: Record<string, string>) {
  setIsGenerating(fieldKey)
  try {
    const res = await fetch('/api/claude', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ prompt: buildPrompt(contextFields), maxTokens: 600 }),
    })
    const data = await res.json()
    if (data.error) throw new Error(data.error)
    ;(setValue as (k: string, v: string) => void)(fieldKey, data.text || '')
  } catch {
    // show error state
  } finally {
    setIsGenerating(null)
  }
}
```

`setValue()` after generation triggers auto-save naturally via react-hook-form's watch-based debounce.

### Pattern: New Section Registration
Two files must be updated when adding a new section to a module:
1. `src/lib/modules.ts` — add entry to `MODULE_SECTIONS[slug]` with fields array
2. `src/app/(app)/modules/[slug]/[section]/page.tsx` — add entry to `SECTION_REGISTRY`

**Critical:** Both files must be updated atomically. Missing either causes a 404 at the route.

### Anti-Patterns to Avoid
- **Dynamic import in SECTION_REGISTRY without file existing:** webpack fails at build time. Create the component file before adding to registry.
- **Storing base64 images in blp_responses:** Supabase JSONB has limits; mood board images must stay in localStorage (D-09 decision).
- **Calling Instagram Scraper API directly from client:** Old app called RapidAPI directly with a hardcoded key. The new app must route any third-party API calls through `/api/` route handlers to keep keys server-side. However, per D-06/D-07, the creator analysis sends the handle + bio to `/api/claude` for AI analysis — there is no Instagram API integration in the new app. The profile fetch (stats, thumbnails) is intentionally dropped; only the AI analysis prompt remains.
- **Forgetting MODULE_SECTIONS update:** New fields not registered in `MODULE_SECTIONS` won't be loaded on mount (the useEffect loops over saved responses, but `defaultValues` must include the field key for react-hook-form to track it).

## Common Pitfalls

### Pitfall 1: Creator Card State Desync with Auto-Save
**What goes wrong:** Creator cards manage complex in-memory state (handle, analysis results, notes). If this state isn't serialized back to a react-hook-form field, auto-save won't persist it.
**Why it happens:** `useAutoSave` debounces on `watch()` changes. If creator data isn't in a watched field, it silently doesn't save.
**How to avoid:** Serialize the entire creator array to `vw_ca_creators` field on every mutation using `setValue()`, which triggers the watch-based save.
**Warning signs:** Data disappears on page refresh.

### Pitfall 2: Section Index Mismatch After Adding New Sections
**What goes wrong:** New sections (Cinematic Content, Content Blueprint) added to `MODULE_SECTIONS` shift the array indices of existing sections. Any component that hardcodes `SECTION_INDEX = N` would point at the wrong section.
**Why it happens:** All components use `MODULE_SECTIONS['content']![SECTION_INDEX]` where `SECTION_INDEX` is a constant.
**How to avoid:** Add new sections to the END of the content module's array in `MODULE_SECTIONS`, not in the middle. Then set `SECTION_INDEX` in the new component files to the correct new index position.
**Warning signs:** Wrong fields loading for a section, or SectionWrapper tracking completion against wrong fields.

### Pitfall 3: Pinterest Embed Script Requires Client-Only Execution
**What goes wrong:** Pinterest embed script (`//assets.pinterest.com/js/pinit.js`) mutates the DOM after load. In React SSR/RSC context, running it on the server crashes.
**Why it happens:** Script relies on `window` and DOM APIs.
**How to avoid:** Load the Pinterest script inside a `useEffect(() => { ... }, [pinUrl])` hook, only when the user has a Pinterest URL saved. Dynamically create and append the `<script>` tag.
**Warning signs:** Hydration errors or `window is not defined` errors.

### Pitfall 4: Google Fonts Live Preview Accumulates `<link>` Tags
**What goes wrong:** Each keypress in the font input fires a new Google Fonts `<link>` tag injection, accumulating duplicate stylesheet loads.
**Why it happens:** Naive implementation appends a new `<link>` on every value change.
**How to avoid:** Use a unique `id` attribute on the injected `<link>` tags (e.g., `gf-primary`, `gf-body`). Before injecting, check if the element already exists and update `href` instead of creating a new element. Debounce with a short delay (300-500ms) since the font name is typed character by character.
**Warning signs:** Network panel shows dozens of duplicate Google Fonts requests.

### Pitfall 5: localStorage Quota Exceeded with Many Mood Board Images
**What goes wrong:** base64 images are ~30-50KB each; 20+ images hits 5-10MB localStorage limit. Save silently fails.
**Why it happens:** localStorage has a ~5MB per-origin limit in most browsers.
**How to avoid:** Wrap all `localStorage.setItem()` calls in try/catch (old app does this). Show a user-visible warning if quota is exceeded. Limit per-category image count to a reasonable cap (e.g., 20 per category, matching old app behavior).
**Warning signs:** Images appear to upload but disappear on next load.

### Pitfall 6: AI Generate Button Without Input Validation
**What goes wrong:** AI generate buttons fire with empty context fields, wasting API tokens and showing confusing empty results.
**Why it happens:** No guard check before calling `/api/claude`.
**How to avoid:** Check that required context fields are non-empty before firing. Show an inline message ("Fill in the fields above first") if validation fails — matches old app's `qh` hint text pattern. Same pattern used in `trust-and-money.tsx` already.
**Warning signs:** Blank AI responses or generic/confused AI output.
